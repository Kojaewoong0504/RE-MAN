# Style Program UX v2

## Purpose

스타일 프로그램은 AI 이미지 생성 체험이 아니라 `지금 가진 옷으로 첫 변화를 시작하게 만드는 코칭 플로우`다.

사용자는 다음을 명확히 이해해야 한다.

- 내가 지금 어떤 상태인지
- 오늘 무엇을 하나만 바꾸면 되는지
- 추천 조합을 어떻게 눈으로 확인할 수 있는지
- 실제 실착 생성이 가능한 상태인지, 단순 레퍼런스 확인인지

---

## Visual Thesis

밝은 오프화이트 배경, 실제 사진 중심의 큰 시각 앵커, 절제된 블랙 타이포, 차분한 남성 스타일 코치 앱.

포스터 실험처럼 모든 영역을 강하게 보이게 하지 않는다. 사진과 추천 조합이 주인공이고, UI는 보조다.

---

## Content Plan

### 1. Landing

역할: 서비스의 약속을 5초 안에 이해시킨다.

필수 요소:

- 브랜드: `RE:MEN`
- 헤드라인: “지금 가진 옷으로 첫인상을 정리한다”
- 설명: 사진 한 장과 옷장 컨텍스트로 오늘 바꿀 한 가지를 찾는다
- CTA: “스타일 체크 시작”

금지:

- 기능 카드 나열을 첫 화면의 중심으로 두지 않는다
- 추상적인 변화 문구를 반복하지 않는다

---

### 2. Program Hub

역할: 확장 가능한 변화 프로그램 구조를 보여준다.

MVP에서는 스타일만 활성화하고, 헤어/체형/피부는 준비 중으로 둔다.

---

### 3. Style Check

역할: 사용자의 현재 상태와 목표를 빠르게 파악한다.

구성:

- 현재 스타일
- 변화 계기
- 예산
- 목표
- 자신감

원칙:

- 사용자가 버튼처럼 보는 선택지는 실제 `button`이어야 한다
- 선택 상태는 저장되고 `/api/feedback` payload에 들어가야 한다
- 질문은 평가처럼 보이면 안 된다

---

### 4. Photo Upload

역할: 분석 가능한 사진과 옷장 컨텍스트를 받는다.

우선순위:

1. 사진 업로드
2. 오늘 입은 옷 텍스트 대체
3. 자주 입는 옷장 컨텍스트

필수 UX:

- 좋은 사진 조건을 짧게 안내한다: 전신, 정면, 밝은 곳
- 사진이 없으면 텍스트 설명으로도 진행 가능하다
- 업로드 후 “사진 준비 완료”가 명확해야 한다

---

### 5. Analysis

역할: 사용자가 기다리는 이유를 이해하게 한다.

진행 문구:

- 핏을 보는 중
- 색 조합을 보는 중
- 지금 가진 옷으로 가능한 조합을 찾는 중

하네스 규칙:

- 개발 모드 StrictMode에서 `/api/feedback`가 중복 호출되면 실패다
- 실제 Gemini 브라우저 업로드 검증은 `npm run smoke:feedback:browser`로 확인한다

---

### 6. Result

역할: 사용자가 결과를 한 번에 이해하고 바로 행동하게 한다.

카드 넘김을 핵심 구조로 쓰지 않는다.

권장 레이아웃:

```text
[업로드한 사진]
[진단 한 줄]
[바꿀 점 3개]
[추천 조합]
[지금 바꿀 행동 1개]
[다음 행동 허브]
[사용자가 선택한 상세 영역]
```

원칙:

- 추천 조합은 숨기지 않는다
- 오늘 할 일은 하나만 제안한다
- 구매 유도는 사용자가 구매 후보를 명시적으로 요청하기 전까지 금지한다
- MVP 결과 화면에는 생성성 보조 기능을 쌓지 않는다
- 남기는 기능은 추천 반응 저장, 새 체크 시작, 기록 보기뿐이다. 옷장 수정은 하단 탭 또는 옷장 화면에서 처리하고, 계정 저장은 분석 완료/반응 저장 시점에 자동 처리한다
- deep-dive, try-on, credits는 기본 프론트에서 숨긴다. 다시 노출하려면 대기 상태와 중복 요청 제어를 먼저 설계한다

---

### 7. Try-On / Reference

역할: 추천 조합을 눈으로 확인한다.

상태를 명확히 분리한다.

- MVP 기본 결과 화면에는 `조합 느낌 보기` 패널을 노출하지 않는다. 레퍼런스/실착 확인은 provider 상태, 중복 요청 제어, 비용 안내, 개인정보 안내를 별도 승인한 뒤 다시 연결한다.
- `TRY_ON_PROVIDER=mock`: 실제 생성 금지, 레퍼런스 확인만 제공
- `TRY_ON_PROVIDER=vertex`: 실제 Vertex 생성 가능, 비용과 개인정보 안내 후 생성

금지:

- mock 결과를 실제 실착 생성처럼 표현하지 않는다
- 레퍼런스 placeholder를 “실착 이미지”라고 부르지 않는다
- MVP 기본 결과 화면에서 `/api/try-on`을 호출하거나 크레딧을 차감하는 진입점을 만들지 않는다

---

## Interaction Thesis

- 업로드한 사진은 결과 화면에서 계속 기준 이미지로 남긴다
- 추천 조합은 한 화면에서 즉시 보이게 한다
- 실착 생성은 MVP 기본 프론트에서 노출하지 않는다. provider 상태가 실제로 열리고 중복 요청 제어가 준비된 뒤 별도 기능으로 다시 연결한다
- 홈, 프로그램, 프로필은 기능 홍보 페이지가 아니라 사용자를 다음 작업으로 보내는 작업 허브다
- 업로드 화면은 사진 입력과 옷장 입력을 동등하게 펼치지 않는다. 사진을 먼저 받고, 옷장은 요약과 편집 작업으로 분리한다

---

## Design Rules

- 큰 보더와 포스터 스타일은 강조 구간에만 제한적으로 사용한다
- 기본 화면은 더 실제 앱처럼: 얇은 구분선, 넓은 여백, 큰 사진, 명확한 CTA
- 카드는 카드 자체가 상호작용일 때만 사용한다
- 다크 테마를 기본값으로 쓰지 않는다
- 기능 설명보다 사용자의 다음 행동을 먼저 보이게 한다
- 한 화면에서 사용자가 누를 수 있는 주요 CTA는 1개, 보조 CTA는 행동 허브로 모은다
- 같은 종류의 보조 기능이 3개 이상이면 카드 그리드가 아니라 리스트/레일로 보여준다

---

## Harness Rules

- mock E2E 통과를 실제 Gemini 또는 Vertex 동작으로 보고하지 않는다
- 실제 Gemini 브라우저 업로드는 `npm run smoke:feedback:browser`로 검증한다
- 실제 Vertex 실착 생성은 `/api/try-on` POST가 `status: "vertex"`와 `preview_image`를 반환해야 성공으로 본다
- provider가 mock이면 생성 버튼은 비활성화되어야 한다
