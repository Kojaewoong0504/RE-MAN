# Style History

## Purpose

사용자는 이전 스타일 체크 결과와 본인의 반응을 다시 확인할 수 있어야 한다.
이 기록은 운영 관제나 장애 대응이 아니라, 사용자의 재방문과 개인화 추천을 위한 제품 기능이다.

## Scope

- 로그인 사용자만 `/history`에서 접근한다.
- 최근 스타일 체크 결과, 추천 조합, 사용자 반응, deep dive 기록, 루틴 기록을 보여준다.
- Firestore에 저장된 프로그램 상태가 있으면 로컬 상태와 병합해 보여준다.
- 기록 화면은 사진 원본이나 실착 생성 이미지를 영구 저장/노출하는 기능이 아니다.

## Non-Negotiables

- mock 또는 로컬 기록을 실제 Gemini 검증 기록으로 표현하지 않는다.
- 사용자가 저장하지 않은 결과를 계정 기록처럼 표현하지 않는다.
- 기록은 사용자가 다음 체크를 이어가기 위한 컨텍스트이지, 운영 중 장애 관제 화면이 아니다.

## UX Rules

- 먼저 최근 추천 조합과 사용자의 반응을 보여준다.
- 로컬 기록이 있으면 원격 동기화보다 먼저 즉시 보여준다. Firestore 병합은 화면을 막지 않는 보강 작업이어야 한다.
- 상단 대표 영역은 로컬 기록이 있는 상태에서 "불러오는 중"으로 오래 남아 있으면 안 된다.
- 원격 기록 로드 실패를 기본 화면에 빨간 오류 카드로 계속 노출하지 않는다. 로컬 기록을 fallback으로 보여주고, 실제 저장 실패는 저장 액션에서만 명확히 알린다.
- 기록은 일별, 주별, 월별 보기로 전환할 수 있어야 한다.
- 기록은 단순 문장 목록이 아니라 클릭 가능한 카드로 보여준다.
- 카드 기본 상태는 제목, 유형, 짧은 요약만 보여주고, 클릭하면 행동/진단/이동 액션을 펼친다.
- 스타일 체크 카드 상세에는 저장된 사용자 반응과 추천 근거 아이템을 함께 보여준다.
- 스타일 체크 카드 상세의 기본 순서는 `추천 조합` -> `오늘 실행` -> `추천에 쓴 옷` -> `내 반응`이다.
- 추천 근거는 `recommended_outfit.source_item_ids`를 우선 사용하고, 없으면 결과 화면과 같은 보수적 매칭 규칙을 따른다.
- 추천 근거 label은 결과 화면과 같아야 한다. `추천에 사용`, `비슷한 후보`, `추가 후보` 외의 내부 상태명을 노출하지 않는다.
- 긴 진단은 카드 안에서 요약 중심으로 보여주고, 자세한 결과는 기존 결과 화면으로 이동하게 한다.
- 스타일 체크 기록은 `비슷하게 다시 체크` 행동을 제공한다. 이 행동은 옷장, 추천 반응, 히스토리를 유지하고 현재 사진/결과/실착 캐시만 비운 뒤 업로드 화면으로 이동한다.
- 하단 탭으로 갈 수 있는 `옷장 관리` 같은 중복 이동은 기록 상세에 반복하지 않는다.
- 옷장 아이템의 착용 빈도와 계절은 다음 분석 payload에 포함되어, 자주 입는 기본템과 계절 후보를 구분하는 근거로 쓴다.
