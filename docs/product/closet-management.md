# Closet Management

## Purpose

RE:MEN의 추천은 사진 평가만으로 끝나면 안 된다.
사용자가 실제로 가진 옷을 기준으로 조합을 제안해야 하므로, 옷장은 MVP의 핵심 입력이다.

## Scope

- 옷장은 현실의 옷장 속에 있는 옷을 사진으로 찍어 저장하는 기능이다.
- 사용자는 전용 옷장 화면에서 옷 사진을 먼저 등록하고, 필요할 때 상의, 하의, 신발, 겉옷 분류와 이름/색/핏/사이즈/착용감/메모를 붙인다.
- 각 아이템은 사진을 기본 입력으로 가진다. 텍스트는 AI 추천에 쓰기 위한 보조 설명이다.
- 저장된 옷장은 다음 스타일 체크 요청의 `closet_items`와 `closet_profile`로 전달된다.
- 로그인 사용자는 Firestore 프로필에 동기화하고, 로컬 상태에도 같은 값을 저장한다.
- 비로그인 사용자는 보호 페이지에 접근하지 못하며 로그인 화면으로 이동한다.
- 로컬 옷장 데이터가 있으면 Firestore 동기화보다 먼저 즉시 보여준다. 원격 프로필 조회는 화면을 막지 않는 보강 작업이어야 한다.
- 사용자가 옷장 편집을 시작한 뒤 도착한 원격 프로필은 현재 편집 중인 아이템을 덮어쓰면 안 된다.

## Non-Negotiables

- 옷장 입력을 단순 장식 UI로 두지 않는다. 저장된 아이템은 반드시 다음 분석 payload에 포함되어야 한다.
- 사용자가 등록하지 않은 옷을 “옷장 기반 추천”으로 표현하지 않는다.
- 옷장 사진을 분석용 전신 사진처럼 `/api/feedback`의 `image`로 섞어 보내지 않는다. 현재 하네스는 1회 API 호출당 분석 이미지 1장만 허용한다.
- 옷장 사진 자체를 AI가 읽었다고 표현하지 않는다. MVP에서 AI가 받는 것은 사진에 붙은 분류/메모/아이템명 요약이다.
- 사진 기반 사이즈 추정과 실제 상품 추천은 별도 기능이다. 옷장 페이지에서는 가진 옷 컨텍스트만 관리한다.

## UX Rules

- 처음 등록은 부담이 낮아야 한다. 옷장 사진 1장과 대략적인 분류만으로 저장 가능하다.
- 옷장 기본 화면은 등록 폼이 아니라 실제 옷장처럼 보이는 compact inventory여야 한다.
- 외곽 프레임, 행거 레일, 선반, 서랍, 신발 칸 같은 옷장 물성을 UI에 반영한다.
- 상의, 하의, 신발, 겉옷은 선반/서랍 단위로 묶어 한눈에 보이게 한다.
- 각 옷장 칸은 기본 닫힘 상태다. 사용자가 해당 칸을 클릭해야 내부 아이템이 펼쳐진다.
- 아이템은 썸네일 중심으로 보여주고, 자세한 메모는 클릭 후 더보기 영역에서 확인한다.
- 새 아이템 등록은 화면 상단의 통합 `+` 버튼 하나로만 연다. 카테고리별 `+` 버튼을 반복 노출하지 않는다.
- 추가할 종류는 등록 모달 안에서 선택한다.
- 피하고 싶은 핏/색/아이템은 옷장 화면에 별도 `Avoid` 입력으로 노출하지 않는다. 필요하면 설정 또는 분석 전 입력에서 관리한다.
- 등록 모달에서도 사진 미리보기가 텍스트 폼보다 먼저 보여야 한다.
- 카테고리별 개수를 보여 사용자가 무엇이 부족한지 바로 알 수 있게 한다.
- 로컬 옷장이 있는 상태에서 상단 카운트가 `0` 또는 `불러오는 중`으로 오래 보이면 안 된다.
- 저장 후 바로 스타일 체크로 이동할 수 있어야 한다.
