# Credits

## 목적

크레딧은 비용이 발생할 수 있는 스타일링 액션을 무제한 호출하지 못하게 막는 사용량 단위다.
MVP v0에서는 결제, 구매, 환불, 구독을 구현하지 않는다.

핵심 AI 기능을 사용할 수 있는 사용자는 아래 중 하나여야 한다.

1. 로그인 세션이 있는 사용자
2. 활성 구독 권한이 있는 사용자
3. 유료 크레딧 또는 이벤트성 크레딧 잔액이 있는 사용자

## v0 범위

- 로그인 사용자는 이벤트성 시작 크레딧을 가진다.
- `/programs/style/onboarding/*`, `/api/feedback`, `/api/try-on`, `/api/credits`는 로그인 세션이 필요하다.
- `/api/credits/transactions`는 로그인 사용자의 최신 크레딧 거래 원장을 반환한다.
- `/api/feedback` 스타일 체크는 활성 구독이 있으면 차감하지 않고, 구독이 없으면 1 크레딧을 차감한다.
- 크레딧 확인용 프론트 페이지는 MVP 기본 화면에서 제거한다. 현재는 서버 계약과 API 보호만 유지한다.
- 실제 Vertex 실착 생성이 가능한 상태에서 `/api/try-on` 생성이 성공하면 1 크레딧을 차감한다.
- `TRY_ON_PROVIDER=mock` 또는 레퍼런스 확인은 실제 생성이 아니므로 크레딧을 차감하지 않는다.
- provider 실패로 스타일 체크 또는 실착 생성이 완료되지 않으면 차감한 크레딧을 복구한다.
- 잔액이 부족하면 비용 발생 요청은 402로 거부한다.
- 분석 화면은 `402 insufficient_credits`를 일반 분석 장애로 표현하지 않고, 크레딧 부족 상태로 안내한다.
- MVP v0에서는 크레딧 구매 플로우가 없으므로 부족 상태에서 구매를 유도하지 않는다.
- 비용이 발생하는 요청은 `Idempotency-Key`를 지원해야 하며, 같은 성공 요청을 재전송해도 추가 차감하지 않는다.

## v0에서 하지 않는 것

- 결제 수단 연결
- 크레딧 구매
- 환불 정책 UI
- 관리자 수동 조정
- 크레딧 양도
- 실제 구독 결제 연동

## UI 원칙

- 크레딧 잔액은 로그인 사용자가 핵심 기능 사용 가능 여부를 이해할 수 있게 앱 헤더 우측 배지로 짧게 노출한다.
- 내 정보 화면은 최근 크레딧 거래 3개를 보여줘 사용자가 잔액 변화 이유를 확인할 수 있어야 한다.
- 크레딧 표시는 숫자와 상태만 보여주고, 결제/구매/충전 CTA는 MVP v0에서 노출하지 않는다.
- mock/레퍼런스 모드에서는 "크레딧을 사용했다"는 표현을 쓰지 않는다.
- 잔액이 부족한 경우 구매를 유도하지 않고 "충전 기능은 준비 중"이라고 안내한다.
- 크레딧 페이지를 다시 만들려면 먼저 실착 생성 UI, 중복 요청 방지, 결제/비용 고지를 별도 문서로 승인한다.

## 데이터 원칙

- 서버가 최종 잔액을 판단한다.
- 클라이언트 잔액 표시는 안내용이다.
- 실제 차감은 `/api/feedback`, `/api/try-on` 서버 route에서만 수행한다.
- 이벤트성 크레딧을 먼저 사용하고, 이후 유료 크레딧을 사용한다.
- 활성 구독 사용자는 스타일 체크 차감 없이 통과한다.
- 크레딧 잔액은 거래 원장으로 설명 가능해야 한다. 지급, 차감, 환불, 구독 사용, 구독 상태 변경은 모두 기록한다.
- 거래 원장은 최신순으로 조회되며 `reason`, `reference_id`, 차감/환불 후 잔액을 포함해야 한다.
- 비용이 발생하는 요청은 `reference_id`를 공유해 차감과 환불을 같은 사건으로 연결해야 한다.
- 비용이 발생하는 요청은 `idempotency_key`를 원장에 기록해야 한다. 같은 `idempotency_key`의 성공 거래가 이미 있으면 기존 차감으로 처리하고 새 차감을 만들지 않는다.
- 실패 후 환불된 요청은 같은 `idempotency_key`로 다시 시도할 수 있어야 한다. 환불된 실패까지 영구 차단하면 사용자가 네트워크/AI 실패를 복구할 수 없다.
- `/api/credits`는 잔액 반환 전 원장 합계와 계정 잔액을 검산해야 한다. 불일치하면 정상 잔액처럼 보여주지 않는다.
- 운영에서 결제/환불을 붙이기 전에는 in-memory 원장을 영속 DB 원장으로 교체해야 한다.
- v0 저장소는 로컬 개발/테스트를 위해 in-memory store를 사용할 수 있다.
  운영 저장소는 Firestore 또는 서버 DB로 교체해야 한다.
