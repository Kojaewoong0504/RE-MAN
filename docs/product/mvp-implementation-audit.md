# MVP Implementation Audit

## 목적

이 문서는 RE:MEN MVP 요구사항과 현재 구현 상태의 차이를 추적한다.
기능을 완료처럼 말하기 전에 문서, 코드, 테스트가 어디까지 맞는지 확인하기 위한 제품 감사 문서다.

## MVP 요구사항별 현재 상태

| 요구사항 | 현재 상태 | 판단 |
|---|---|---|
| 전신 사진 기반 스타일 추천 | 사진 또는 텍스트 입력 후 `/api/feedback`으로 Gemini/mock 피드백을 생성한다. 결과에는 진단, 개선 포인트, 추천 조합, 오늘 행동이 있다. | 부분 구현 |
| 옷장 등록 기반 조합 추천 | 업로드/옷장/설정 화면에서 현실 옷장 사진을 등록하고, 사진에 붙인 상의/하의/신발/겉옷 분류, 착용감, 착용 빈도, 계절, 상태, 메모를 분석 payload의 카테고리별 요약, 아이템 id, `closet_strategy`로 포함한다. 로그인 사용자의 옷장 아이템은 서버 route를 통해 `users/{uid}/closetItems/{itemId}` 서브컬렉션에 저장하고, 사진은 Supabase Storage 메타데이터로 관리한다. 사진 원본은 현재 `/api/feedback` payload에 보내지 않는다. `2026-04-16` 실제 Gemini smoke에서 `source_item_ids` 반환을 확인했다. | 기본 구현 |
| 의상/사용자 사이즈 추정 기반 상품 추천 | 사용자 입력 사이즈와 옷장 아이템의 실제 사이즈/착용감/상태를 기준으로 `사이즈 체크 후보`를 만든다. 내부 카탈로그 후보는 접힌 보조 패널 안에서만 참고용으로 표시하며, 사진 기반 추정, 외부 상품 카탈로그, 가격/재고/구매 링크는 없다. | 부분 구현 |
| 추천 피드백 저장/재확인/개인화 | 로그인 후 결과, 설문/옷장 정보, 추천 반응을 Firestore에 저장하고 프로필/기록에서 저장된 피드백 타임라인을 복원한다. 기록 화면은 저장 반응, 추천에 쓴 옷, 다음 행동을 펼침 카드로 재확인할 수 있고, `비슷하게 다시 체크`로 옷장/반응/히스토리를 유지한 새 분석을 시작할 수 있다. 새 사진 체크에는 compact `feedback_history`와 구조화된 `preference_profile`이 payload로 전달된다. | 기본 구현 |
| 스타일링 크레딧 | 서버 API와 실제 Vertex 실착 성공 시 차감 계약은 있으나, `/credits` 프론트 페이지와 결과 화면 진입점은 MVP 기본 프론트에서 제거했다. | 보류 |
| 소셜 로그인 | Firebase Google 로그인, 서버 JWT 세션, access/refresh token, RTR refresh route, 보호 페이지가 있다. | 기본 구현 |
| 헤어/피부/향수 등 확장성 | 프로그램 허브와 준비 중 페이지 구조는 있다. 향수는 아직 메뉴/문서 표현이 약하고 실제 프로그램 계약은 없다. | 구조만 구현 |
| UI/UX 완성도 | 랜딩, 프로그램 허브, 스타일 체크, 결과, 프로필, 설정 흐름은 있다. 결과 화면은 추천 조합, 근거 옷, 오늘 행동, 반응 저장 중심으로 정리했다. 다만 화면 증거 기반 검증과 실제 사용자 플로우 검증은 계속 필요하다. | 개선 중 |

## 현재 MVP에서 부족한 핵심

1. 옷장은 사진 기반 등록과 `closet_strategy` 분류로 추천 우선순위를 줄 수 있고, 착용 빈도/계절/상태는 입력받는다. 로그인 사용자 저장은 서브컬렉션/Storage 메타데이터 구조로 이동했다. 아직 사진 자동 판독은 없다.
   MVP에서는 충분하지만 사이즈/상품 추천 단계 전에는 더 정교한 모델이 필요하다.

2. 추천 결과 화면은 `recommended_outfit.source_item_ids`가 있으면 이를 우선 사용하고, 없으면 추천 조합 텍스트와 옷장 아이템을 보수적으로 매칭한다.
   `npm run smoke:feedback:gemini`는 등록된 옷장 아이템 id가 `source_item_ids`로 돌아오는지 검사한다. `2026-04-16` 로컬 `AI_PROVIDER=gemini` smoke에서 상의/하의/신발 id 반환을 확인했다.

3. 사이즈 추천은 결과 화면의 MVP 기본 경로에서 제거했다.
   사진 기반 추정, 외부 상품 카탈로그 추천, 가격/재고/구매 링크는 아직 구현하지 않는다.

4. 크레딧은 v0 사용량 제한으로만 구현되어 있다.
   결제, 충전, 환불, 운영 저장소는 아직 없다.

5. `day1_mission`과 Routine Mode 잔재가 일부 코드/테스트/문서에 남아 있다.
   기본 경험은 Style Check Session이고 루틴은 보조 모드라는 원칙을 계속 강화해야 한다.

6. UI/UX 변경의 완료 보고에는 스크린샷이나 visual smoke 산출물이 필요하다.
   주요 화면 배치 변경은 `npm run visual:app`으로 홈, 스타일, 업로드, 분석 오류, 결과, 옷장, 기록, 기록 상세, 내 정보, 설정 화면을 함께 확인한다.
   하네스 기준상 테스트 통과만으로 UI 완성도를 주장하면 안 된다.

## 다음 구현 우선순위

1. 다음 단계에서는 옷장 사진 자동 판독을 도입할지, 아니면 수동 입력 기반 MVP를 먼저 안정화할지 결정한다.
2. 크레딧 구매/결제/환불은 실제 결제 provider 선택 후 별도 단계로 진행한다.
3. 헤어/피부/향수는 프로그램 허브의 확장 슬롯으로 유지하되, 스타일 MVP가 안정되기 전 실제 코칭 API는 만들지 않는다.

## 검증 기준

- mock E2E 통과는 실제 Gemini 분석 완료로 보고하지 않는다.
- 실제 사진 분석은 `npm run smoke:feedback:gemini`와 필요한 경우 `npm run smoke:feedback:browser`로 분리 검증한다.
- UI/UX 변경은 Playwright 스크린샷 또는 visual smoke 산출물을 남긴다.
  여러 화면에 걸친 변경은 `output/playwright/app-visual-smoke/report.json`까지 함께 확인한다.
- 저장/개인화 변경은 단위 테스트, 통합 테스트, E2E 중 최소 두 계층 이상에서 확인한다.
