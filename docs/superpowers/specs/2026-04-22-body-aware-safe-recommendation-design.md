# Body-Aware Safe Recommendation Design

## 1. Goal

RE:MAN의 스타일 추천을 "보기 좋은 모델 기준 추천"이 아니라 "현재 사용자 체형, 현실 옷장, 현재 예산 안에서 실패 확률이 가장 낮은 안전한 추천"으로 재정의한다.

이번 설계의 핵심은 다음 두 가지다.

1. 추천의 중심을 `멋있어 보이는 후보 찾기`에서 `망할 확률이 높은 후보 제거하기`로 바꾼다.
2. 사용자의 체형 신호를 직접 평가하거나 낙인찍지 않고, 시각적 보정 관점의 구조화 신호로만 사용한다.

이 설계는 MVP 핵심 경로인 Style Check Session에 적용하며, 7일 루틴이나 실착 생성보다 먼저 추천 품질 기준을 바로잡는 데 목적이 있다.

---

## 2. Product Direction

이번 기능의 제품 방향은 아래처럼 고정한다.

- 우선순위: `현실성 우선`
- 추천 성격: `안전한 개선형`
- 기준 사용자: 체형이 크거나 비율 보정이 필요한 사용자도 현재 옷장 기준으로 바로 실행 가능한 추천을 받는다.

즉 시스템은 다음 질문에 답해야 한다.

> "이 사용자가 지금 가진 옷 안에서 덜 망하고, 조금 더 정리돼 보일 가능성이 가장 높은 조합은 무엇인가?"

반대로 아래 방향은 이번 설계의 목적이 아니다.

- 모델 체형을 가정한 이상적 스타일 제안
- 구매 전제를 깔고 시스템 추천을 메인으로 미는 흐름
- 사용자의 체형을 직접 평가하거나 숫자로 단정하는 진단

---

## 3. Recommendation Principle

추천 엔진은 아래 순서를 따른다.

1. 사진에서 체형/비율 관련 시각 신호를 추출한다.
2. 실패 회피 규칙을 먼저 적용해 위험한 후보를 제거한다.
3. 남은 옷장 후보를 점수화해 가장 안전한 조합을 고른다.
4. 안전한 후보가 부족할 때만 시스템 추천으로 보강한다.

이때 성공 기준은 "추천이 화려한가"가 아니라 아래와 같다.

- 현재 체형에서 시선이 과하게 몰리는 요소를 줄였는가
- 현재 옷장 안에서 바로 재현 가능한가
- 사용자가 읽었을 때 판단당하는 느낌 없이 이유를 이해할 수 있는가

---

## 4. New Input Contract

기존 `survey`, `closet_items`, `closet_strategy`, `image`에 더해 추천 전용 구조화 신호 `body_profile`을 추가한다.

```json
{
  "body_profile": {
    "upper_body_presence": "high",
    "lower_body_balance": "medium",
    "belly_visibility": "high",
    "leg_length_impression": "shorter",
    "shoulder_shape": "rounded",
    "neck_impression": "short",
    "overall_frame": "large",
    "fit_risk_tags": [
      "tight_top_risk",
      "cropped_top_risk",
      "strong_contrast_split_risk"
    ]
  }
}
```

### 4.1 Field Intent

- `upper_body_presence`
  - 상체 존재감이 큰지, 상체 시선 집중 위험이 큰지 표현한다.
- `lower_body_balance`
  - 하체 쪽 시각 균형이 부족한지 여부를 나타낸다.
- `belly_visibility`
  - 복부 쪽 시선이 모이는 정도를 정성적으로 나타낸다.
- `leg_length_impression`
  - 다리 비율이 짧아 보이는지, 보통인지, 길어 보이는지 표현한다.
- `shoulder_shape`
  - 둥근 어깨, 좁은 어깨, 보통처럼 추천 보정에 영향을 주는 형태만 다룬다.
- `neck_impression`
  - 목 주변이 답답해 보이는지 여부를 나타낸다.
- `overall_frame`
  - 전체 프레임을 `large`, `medium`, `compact` 같은 수준으로만 구분한다.
- `fit_risk_tags`
  - 추천 실패를 유발하는 금지 조건 태그 집합이다.

### 4.2 Constraints

- 숫자 기반 체형 추정은 하지 않는다.
- 체중, 허리둘레, BMI 같은 민감 추정값은 만들지 않는다.
- body profile은 "정밀 진단 데이터"가 아니라 "실패 회피용 추천 힌트"다.
- 사용자의 체형을 비하하거나 의료적 상태처럼 다루는 표현을 금지한다.

## 5. New Output Contract

추천 결과는 기존 `recommended_outfit`를 유지하되, 왜 이 추천이 "안전한지"를 설명하는 구조를 추가한다.

```json
{
  "recommended_outfit": {
    "title": "안전한 기본 조합",
    "items": ["검정 레귤러 티셔츠", "차콜 테이퍼드 팬츠", "검정 운동화"],
    "reason": "상체 부각을 과하게 키우지 않고 하체 라인을 정리하는 조합",
    "safety_basis": [
      "상하 대비가 과하지 않음",
      "허벅지와 종아리 라인이 과도하게 드러나지 않음",
      "현재 체형에서 가장 무난한 실루엣"
    ],
    "avoid_notes": [
      "짧은 기장의 상의는 피함",
      "밝은 하의로 시선이 쏠리는 조합은 제외"
    ],
    "try_on_prompt": "상체 존재감을 과하게 키우지 않고 하체 비율이 더 안정적으로 보이도록 전체 조합을 자연스럽게 적용"
  }
}
```

### 5.1 Output Rules

- `reason`은 2줄 이내의 짧은 설명으로 제한한다.
- `safety_basis`는 사용자가 납득할 수 있는 짧은 근거 2~3개를 반환한다.
- `avoid_notes`는 긴 문장이 아니라 짧은 경고 라벨로 변환 가능한 형태여야 한다.
- 추천은 항상 1개의 기본 조합만 메인으로 제시한다.
- 시스템 추천은 메인 조합을 대체하지 않고 보강용 reference로만 노출한다.

## 6. Decision Flow

### 6.1 Step 1: Extract Visual Risk Signals

사진 분석 단계에서는 체형을 평가하지 않고 추천 위험 신호만 추출한다.

예시 신호:

- 상체 존재감이 크다
- 복부 시선 집중이 크다
- 다리가 짧아 보인다
- 목 주변이 답답해 보인다
- 어깨 라인이 둥글게 보인다

이 단계의 산출물은 "어떤 옷이 어울리는가"가 아니라 "어떤 조합이 위험한가"를 위한 입력이다.

### 6.2 Step 2: Apply Failure-Avoidance Rules

다음과 같은 보수적 제외 규칙을 기본으로 둔다.

- 상체 존재감이 크면 너무 얇고 몸에 붙는 상의 제외
- 복부 부각이 크면 짧은 기장 상의 제외
- 다리가 짧아 보이면 상하 대비가 강한 조합 제외
- 목이 짧아 보이면 답답한 넥라인 제외
- 큰 프레임인데 하의가 지나치게 좁거나 끊겨 보이면 제외

이 단계에서 중요한 것은 무엇을 골랐는지보다 무엇을 버렸는지다.

### 6.3 Step 3: Score Closet Candidates

실패 회피를 통과한 옷장 후보에 점수를 부여한다.

점수 요소:

- `closet_strategy.role`
- 착용 빈도
- 잘 맞음 여부
- 계절 적합성
- 상태
- body profile과의 충돌 여부
- 상의/하의/신발 연결 안정성

우선순위는 `core > optional > use_with_care`다.
단, `use_with_care`는 다른 대안이 없을 때만 보조 후보로 남긴다.

### 6.4 Step 4: Add System Support Only If Needed

옷장 안에서 안전한 축을 못 만들 때만 시스템 추천을 붙인다.

시스템 추천 기준:

- 멋진 포인트 아이템보다 안정적인 보정 아이템 우선
- 큰 체형에 과한 장식/극단적 실루엣 추천 금지
- 구매 유도 문구 금지
- 메인 블록이 아니라 `보강 참고` 블록으로만 노출

## 7. User Experience

정교한 판단을 하더라도 결과 화면은 더 단순해야 한다.

### 7.1 Result Structure

결과 화면 메인 구조는 4블록으로 제한한다.

1. 한 줄 진단
2. 오늘의 안전한 조합
3. 피해야 할 것
4. 오늘 바로 할 것

### 7.2 Copy Rules

- 한 줄 진단: 2줄 이내
- 추천 이유: 2줄 이내
- 피해야 할 것: 칩 또는 짧은 라벨 2~3개
- 오늘 행동: 1문장

### 7.3 Tone Rules

허용 예시:

- "상체 쪽 시선이 먼저 모여서 하체 균형을 같이 잡아주는 조합이 안전합니다."
- "전체적으로 편한 방향은 맞지만, 비율이 끊겨 보여 조금만 정리해도 훨씬 나아집니다."

금지 예시:

- "체형상 이 옷은 안 어울립니다."
- "뚱뚱해 보여서 피해야 합니다."
- "비율이 이상합니다."

### 7.4 Positioning Rule

- 메인 추천은 항상 `내 옷장 기준`
- 시스템 추천은 항상 `보강용 참고`
- 사용자가 앱을 "쇼핑 유도 서비스"로 느끼게 만들면 실패

## 8. Testing Strategy

이 기능은 추천 문구가 좋아 보인다고 끝나지 않는다. 최소 4층 검증을 요구한다.

### 8.1 Contract Tests

검증 내용:

- 추천 엔진 입력에 `body_profile`이 실제로 포함되는가
- 결과 응답에 `safety_basis`, `avoid_notes`가 포함되는가
- 계약 누락 시 조용히 무시하지 않고 실패를 드러내는가

### 8.2 Rule Tests

fixture 예시:

- body profile: 큰 프레임, 복부 부각, 짧아 보이는 하체
- closet items:
  - 검정 레귤러 티셔츠
  - 차콜 팬츠
  - 검정 운동화
  - 짧은 밝은 상의
  - 슬림 하의

기대 결과:

- 짧은 밝은 상의 제외
- 슬림 하의 제외
- 어두운 안정 조합 우선

규칙 테스트는 "선택 성공"보다 "위험 후보 제거 성공"을 검증한다.

### 8.3 Integration Tests

검증 내용:

- `/api/feedback`가 body-aware rule을 거친 최종 추천을 반환하는가
- verified source item ids와 실제 추천 아이템이 일치하는가
- 시스템 추천이 부족한 카테고리 보강에만 쓰이는가

### 8.4 E2E / UX Tests

브라우저 기준 검증:

- 메인 블록이 `내 옷장 기준`으로 보이는가
- 한 줄 진단이 2줄 내에서 읽히는가
- `피해야 할 것`이 짧은 UI 형태로 보이는가
- 오늘 행동이 구매 없이 가능한가
- 시스템 추천이 메인보다 더 크게 보이지 않는가

---

## 9. Harness Updates Required

이번 설계를 반영하며 AGENTS 또는 verification 문서에 아래 실패 기준을 추가해야 한다.

- `body_profile` 없이 체형 맞춤 추천이라고 보고하면 실패
- 실패 회피 규칙 테스트 없이 추천 정교화 완료라고 보고하면 실패
- 시스템 추천이 메인 블록을 대체하면 실패
- 사용자 체형을 직접 평가하거나 비하하는 문구가 결과 문구에 들어가면 실패
- 오늘 행동이 구매, 복잡한 등록, 다단계 미션이면 실패
- 결과 화면이 추천보다 장문 설명 중심이 되면 실패

---

## 10. Out of Scope

이번 설계에 포함하지 않는 것:

- 체중/치수 기반 정량 추정
- 의료적 자세 분석
- 구매 추천 엔진 고도화
- 루틴 에이전트 전체 재설계
- 실착 provider 개선 자체

이번 설계의 범위는 `추천 판단 기준과 결과 표현 방식 재정의`다.

---

## 11. Recommended Implementation Order

1. 계약 확장: `body_profile`, `safety_basis`, `avoid_notes`
2. 규칙 엔진 추가: failure-avoidance layer
3. 옷장 점수화와 body-aware scoring 결합
4. 결과 화면 간소화
5. unit / integration / e2e 검증 추가
6. 하네스 문서 업데이트

---

## 12. Success Criteria

이 설계가 제대로 구현되면 아래 결과가 보여야 한다.

- 큰 체형/비율 보정이 필요한 사용자가 봐도 "모델 기준 추천"이 아니라는 느낌을 받는다.
- 추천이 더 화려해지지 않아도, 덜 망하는 쪽으로 안정된다.
- 시스템 추천은 메인을 밀어내지 않고 부족한 부분만 메운다.
- 결과 화면에서 사용자는 "왜 이 조합이 안전한지"를 짧게 이해할 수 있다.
