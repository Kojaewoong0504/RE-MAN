# HARNESS_ENGINEERING.md

## Purpose

이 문서는 이 저장소에서 하네스 엔지니어링을 어떻게 정의하고 운영하는지 설명한다.
목표는 AI가 실수하지 않기를 기대하는 것이 아니라, 실수가 반복될 수 없도록 구조를 설계하는 것이다.
이 구조의 초점은 개발 중 바이브 코딩 품질 통제다.
운영 환경의 장애 관제나 프로덕션 모니터링을 하네스의 핵심 책임으로 두지 않는다.

## Core Idea

하네스 엔지니어링의 핵심은 부탁이 아니라 강제다.

- 사람이 잘 짜달라고 요청하는 구조가 아니다.
- AI가 잘못 짜면 자동 검사에 걸리고, 다시 수정하게 만드는 구조다.
- 한 번 발생한 실수는 규칙과 하네스로 승격해 재발을 막아야 한다.

## Scope Boundary

하네스가 직접 다루는 것:

- 문서 없는 구현 차단
- 구조 규칙 위반 차단
- 타입, 린트, 빌드, 테스트 실패 차단
- 반복 실수의 규칙 승격
- 개발 중 코드베이스 드리프트와 나쁜 패턴 확산 방지
- 테스트 계층의 편중 방지

하네스가 직접 대체하지 않는 것:

- 운영 중 서비스 장애 관제
- 프로덕션 incident 대시보드
- 모니터링, 알림, APM, 로그 검색 시스템

운영 환경의 장애 대응은 Sentry, Vercel, Firebase, Supabase 같은 별도 운영 도구가 맡는다.

## The Five Axes

### 1. Agent Instructions

`AGENTS.md`는 AI 업무 지침서다.

- 어떤 문서를 먼저 읽어야 하는지 정의한다.
- 해도 되는 것과 절대 하면 안 되는 것을 정의한다.
- 문서 없는 기능 추가를 금지한다.

### 2. Gates

저장, 커밋, 푸시, 머지 단계마다 자동 게이트를 둔다.

- 성공은 조용히 지나간다.
- 실패만 짧고 분명하게 알린다.
- 실패한 변경은 다음 단계로 넘어갈 수 없다.
- e2e만으로 검증을 끝내지 않고, 로직은 단위/통합 테스트로 먼저 잡는다.

### 3. Tool Boundaries

AI가 접근할 수 있는 범위와 권한을 미리 제한한다.

- 수정 가능한 경로
- 실행 가능한 명령
- 금지된 파괴적 작업
- 민감 데이터 접근 제한

### 4. Feedback Loops

AI는 작성 후 바로 검사를 받고, 실패하면 스스로 수정해야 한다.

- 작성
- 검사
- 실패 원인 확인
- 재수정
- 재검사

이 루프는 사람 개입 없이 반복될 수 있어야 한다.

현재 저장소에서는 이 루프의 최소 형태로 아래를 둔다.

- `scripts/run-save-gate.sh`
- `scripts/run-self-repair-loop.sh`

첫 번째는 강제 검사고, 두 번째는 실패를 구조화해 다음 수정 입력으로 넘기기 위한 최소 루프다.

### 5. Garbage Collection

AI는 기존 코드를 보고 따라하기 때문에, 나쁜 패턴이 누적되면 코드베이스 전체가 오염된다.
그래서 주기적으로 코드베이스를 청소하는 별도 축이 필요하다.

garbage collection의 목표는 다음과 같다.

- 문서와 코드의 드리프트 제거
- 규칙 위반 코드 제거
- 사용하지 않는 코드 제거
- 중복되거나 임시적인 나쁜 패턴 제거

## System Rule

실패는 티켓으로만 끝나면 안 된다.
재발 가능성이 있는 실패는 반드시 아래 중 하나로 승격해야 한다.

1. `AGENTS.md` 규칙 추가
2. gate 추가
3. harness 체크 추가
4. garbage collection 체크리스트 추가

## Testing Principle

하네스에서 말하는 테스트는 하나가 아니다.
좋은 저장소는 아래 세 계층을 함께 가진다.

- 단위 테스트: 순수 로직, 계약, 경계 조건 검증
- 통합 테스트: route, storage, auth, provider 연결 검증
- e2e 테스트: 실제 사용자 흐름 검증

e2e만 많은 상태는 하네스 관점에서 불완전하다.
특히 인증, 세션, 토큰, 라우트, 규칙 엔진처럼 결정론적인 로직은 단위/통합 테스트로 먼저 고정해야 한다.

## Operating Principle

좋은 하네스는 AI를 더 똑똑하게 만들지 않는다.
대신 틀린 행동이 통과하지 못하게 만든다.

이 저장소는 모델의 선의보다 구조의 강제를 더 신뢰한다.
그리고 그 구조의 주 무대는 운영 중인 서비스가 아니라 개발 중인 저장소다.
