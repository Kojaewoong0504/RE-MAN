# CONSTITUTION.md

## 1. Purpose

이 저장소는 목표로 하는 앱을 설계, 구현, 검증한다.
초보자가 읽고 따라갈 수 있어야 한다.

## 2. Core Values

### 간결성

복잡하면 쪼개고 제거한다.

### 검증 가능성

문서, 코드, 테스트는 서로 일치해야 한다.
테스트는 단일 방식에 의존하지 않고, 단위 테스트, 통합 테스트, e2e 테스트를 역할에 맞게 함께 사용해야 한다.

### 안전성

사용자 데이터는 보호 대상이다.

## 3. Non-Negotiables

- 문서 없이 기능을 추가하지 않는다.
- 테스트를 속여서 통과시키지 않는다.

## 4. Authority Order

저장소에서 판단이 충돌할 때 우선순위는 아래와 같다.

1. `CONSTITUTION.md`
2. 해당 기능을 직접 설명하는 기능 문서
3. 코드
4. 테스트

테스트가 통과하더라도 상위 문서와 충돌하면 올바른 상태로 보지 않는다.

## 5. Documentation Rule

새 기능이나 동작 변경은 먼저 문서에 정의되어 있어야 한다.

- 기능 문서 없이 기능을 추가하지 않는다.
- 구현을 시작하기 전에 해당 기능의 목적, 입력, 출력, 제약이 문서에 있어야 한다.
- 구현 후 문서와 코드가 다르면 코드가 아니라 문서를 먼저 점검하고 맞춘다.

기능 문서의 기준은 다음과 같다.

- 전역 원칙은 `CONSTITUTION.md`가 정의한다.
- 기능별 상세 동작은 해당 기능을 직접 설명하는 문서가 정의한다.

## 6. Data Safety Minimum

사용자 데이터는 기본적으로 보호 대상이며, 최소 기준은 아래와 같다.

- 민감 데이터는 불필요하게 저장하지 않는다.
- 저장이 필요하면 목적과 보관 범위를 문서에 먼저 적는다.
- 삭제가 필요한 데이터는 삭제 시점과 경로가 문서에 있어야 한다.
- 테스트와 디버깅을 이유로 실제 사용자 데이터를 우회 사용하지 않는다.

## 7. Harness Scope

repository harness와 application harness의 책임은 구분한다.
하네스의 1차 목적은 개발 중 바이브 코딩 과정에서 생기는 실수, 드리프트, 규칙 위반을 조기에 막는 것이다.
하네스는 운영 중인 서비스의 장애 관제 시스템을 대체하지 않는다.
운영 환경의 모니터링과 알림은 별도의 운영 도구가 담당한다.

### Repository Harness

repository harness는 저장소 수준의 계약을 검사한다.

- 헌법과 필수 문서의 존재
- 문서 간 핵심 계약의 일치
- 구현 전에 필요한 문서 정의가 존재하는지 여부

### Application Harness

application harness는 앱 수준의 동작을 검사한다.

- 코드가 문서의 입력/출력 계약을 지키는지
- 런타임 동작이 문서의 제약을 지키는지
- 테스트가 실제 동작을 정직하게 검증하는지
- 단위 테스트, 통합 테스트, e2e 테스트가 서로 다른 책임을 분담하는지

### Scope Boundary

하네스에 우선 포함할 대상:

- 개발 중 실수 방지
- 저장/커밋/푸시 전 구조 위반 차단
- 반복 실수의 규칙 승격
- 문서와 코드 드리프트 방지

하네스의 우선 대상이 아닌 것:

- 운영 장애 대시보드
- 실서비스 알림 관제
- 프로덕션 incident 운영 절차
- 모니터링 SaaS를 대체하는 기능

## 8. Open Questions

아래 항목은 아직 합의되지 않았으므로 헌법에 포함하지 않았다.

- repository harness가 검사해야 하는 "핵심 계약"의 최소 집합은 무엇인가
- 기능 문서의 형식을 템플릿으로 강제할 것인가
- 테스트를 속이는 행위의 판정을 어떤 수준까지 자동화할 것인가
